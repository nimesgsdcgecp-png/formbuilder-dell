FormBuilder3

A full-stack enterprise form management platform — drag-and-drop builder, dynamic storage, approval workflows, role-based access, and a public shareable form engine.

Next.js 14 Spring Boot 3 PostgreSQL 17 Session Auth Zustand dnd-kit JPA/Hibernate BCrypt

System Architecture

Three-tier architecture: Next.js frontend ↔ Spring Boot REST API ↔ PostgreSQL database

Browser (Next.js)
→ HTTP/JSESSIONID →
Spring Boot API :8080
→ JPA →
PostgreSQL :5432

Key Architectural Decisions

  • Session-based auth — Uses JSESSIONID cookies with SameSite=Lax. Not JWT. Session timeout is 15 minutes. Max 1 concurrent session per user.
  • Dynamic DDL — When a form is published, DynamicTableService executes CREATE TABLE DDL at runtime to create a dedicated submissions table per form (e.g. sub_my_form_v1). No ORM entity — raw JdbcTemplate with SQL injection guards.
  • Form Versioning — Every publish creates a form_versions snapshot. Submissions are forever tied to the exact version that was active when they were collected.
  • Rule Engine — IF→THEN logic rules are stored as JSON in form_versions.rules and evaluated server-side by RuleEngineService on every submission. Prevents invalid data from reaching the DB.
  • Feature flags — Workflow and LevelUp features are gated via feature.workflow.enabled=true in application.properties.

Core Services Deep Dive

Business logic responsibilities

FormService

The largest service (~900 lines). Handles form CRUD, publish lifecycle, and validation. Key responsibilities:

  • Validates form code with regex ^[a-z][a-z0-9_]{0,99}$ and checks reserved SQL keywords
  • On publish: calls DynamicTableService.createDynamicTable() or alterDynamicTable()
  • Sets code_locked = true on first publish to prevent table renames
  • Generates public_share_token (UUID) on publish
  • Enforces FormBuilderLimitsConfig: max 50 fields, 100 validations, 10 pages
  • Contains duplicate validateFormCode() overloads — one for create, one for update with exclude

SubmissionService

Handles the full submission lifecycle. Chain of responsibility on submit:

Payload size ≤ 100KB
→
Form is PUBLISHED
→
Version match
ExpressionEvaluator
→
RuleEngine VALIDATE
→
Dynamic INSERT
Workflow trigger
→
Email SEND_EMAIL rules
→
Return submissionId

DynamicTableService

Executes raw DDL via JdbcTemplate. Uses parameterized queries for DML but builds DDL with string concatenation (field keys are validated via SqlKeywordValidator before reaching here). Key methods:

  • createDynamicTable() — CREATE TABLE IF NOT EXISTS with standard columns + form fields
  • alterDynamicTable() — ADD COLUMN for new fields added in a re-publish
  • getSubmissions() — paginated SELECT with dynamic WHERE clauses and ORDER BY
  • getColumnValues() — SELECT DISTINCT for LOOKUP field dropdowns
  • repairLongFieldKeys() — utility to fix 63-char PostgreSQL limit violations

RuleEngineService

Evaluates stored JSON rules against submitted data. Supports nested ConditionGroups with AND/OR logic. Triggered actions: VALIDATION_ERROR (throws HTTP 400), REQUIRE (checks field not empty), SEND_EMAIL (triggers async email).

ExpressionEvaluatorService

Parses and evaluates custom validation expressions (e.g. field_a > field_b). Used for field_validations table entries before insertion. Has its own lexer/parser — not using Groovy or SpEL.

WorkflowService

State machine for the approval workflow. A WorkflowInstance has N WorkflowSteps. Steps are evaluated sequentially. Approval advances current_step_index. Full approval → form status becomes PUBLISHED. Rejection → form status becomes REJECTED.

Role-Based Access

Roles are loaded as Spring Security GrantedAuthority objects prefixed with ROLE_. Method-level security uses @PreAuthorize. The CustomPermissionEvaluator handles hasPermission() checks against the permissions table.

🔴 Security Issue: application.properties contains a hardcoded Gmail SMTP password in plaintext. This must be moved to environment variables before any deployment.
⚠️ CSRF disabled — The comment in SecurityConfig says this is intentional for a decoupled SPA. This is acceptable because the session cookie has SameSite=Lax by default on modern browsers, but should be documented explicitly.

Rule Engine

Server-side IF→THEN logic evaluation on every form submission

Rule Structure (stored as JSON in form_versions.rules)

{ "id": "uuid", "name": "Show GitHub field for Engineers", "conditionLogic": "AND" // OR | AND, "conditions": [ { "field": "department", // field_key "operator": "EQUALS", // EQUALS|NOT_EQUALS|GT|LT|CONTAINS "value": "Engineering", "valueType": "STATIC" // STATIC | FIELD } ], "actions": [ { "type": "SHOW", "targetField": "github_url" }, // Frontend only { "type": "REQUIRE", "targetField": "github_url" }, // Backend validated { "type": "VALIDATION_ERROR", "message": "..." },// HTTP 400 { "type": "SEND_EMAIL", "message": "user@example.com" } // Async ] }

Frontend-only actions (SHOW, HIDE, ENABLE, DISABLE) are evaluated in FormRenderer.tsx at fill time but NOT enforced server-side. Only REQUIRE and VALIDATION_ERROR are enforced on the backend.

Multi-step approval chain for form publishing — gated by feature.workflow.enabled

Frontend: Approvals Pages

  • /approvals — Shows pending steps for the current user (approve/reject actions)
  • /approvals/history — Historical view of all handled workflows

Level Up System

A USER role user can request promotion to BUILDER role via POST /level-up/request. An ADMIN reviews pending requests at GET /level-up/pending and decides via POST /level-up/{id}/decide. On APPROVE, the BUILDER role is globally assigned via RoleService.assignRole().

Auto-derived columnName

When a field label changes, useFormStore.updateField() auto-derives the columnName as snake_case (e.g. "First Name" → first_name). All existing rules referencing the old name are automatically updated in the store to prevent broken logic.

Drag-and-Drop (dnd-kit)

Uses DndContext with PointerSensor and closestCorners collision detection. Dragging from Sidebar (new field) and reordering on Canvas (existing fields) are handled separately in handleDragEnd. Nested fields (children of SECTION_HEADER) have their own sortable context.

Key Components

Non-page reusable components

FormRenderer.tsx (~65KB — largest file)

The universal form fill component. Used in /f/[token] (public) and /builder/preview. Renders all 21 field types, evaluates SHOW/HIDE/REQUIRE/ENABLE/DISABLE rules client-side, handles draft saving, and manages multi-page navigation via PAGE_BREAK fields.

PropertiesPanel.tsx (~40KB)

The right panel in the builder. Shows different controls for each field type: options editor (Dropdown/Radio/Checkbox), grid row/col editor, lookup config, calculation formula input, validation constraints (min/max/pattern/minLength/maxLength), and toggle flags (required, readonly, hidden, disabled, multiselect).

RouteGuard.tsx

Wraps protected pages. Calls GET /auth/me on mount. Redirects to /login if 401. Passes through if authenticated. Does not check specific permissions — that is done by usePermissions inside each page.

Sidebar.tsx (app-level)

The left navigation sidebar. Renders modules fetched from GET /menu based on the user's role-module assignments. Includes user info and logout button. Collapses on mobile.

State Management

Zustand stores — no Redux, no Context API

useFormStore

The single source of truth for the entire builder. Key state:

  • schema: FormSchema — full form including title, code, fields[], rules[], formValidations[]
  • selectedFieldId: string | null — which field card is selected (drives PropertiesPanel)
  • isThemePanelOpen: boolean

Key actions: addField, updateField (with auto-columnName + rule update), removeField, reorderFields, addRule, updateRule, deleteRule, setFormValidations.

Uses recursive updateFieldInTree() for nested children fields.

useUIStore

Simple store: isSidebarOpen: boolean. Controls the collapsible sidebar state.

usePermissions hook

Not Zustand — uses React state + module-level cache. Fetches GET /auth/permissions and caches the result for 5 minutes. Provides hasPermission(permission, formId?) for component-level permission checks. Used on almost every protected page to show/hide action buttons.

Bug Fixes

Issues found that cause incorrect behavior

Bug Duplicate validateFormCode() logic — inconsistent reserved keyword check
FormService defines two nearly-identical overloads of validateFormCode(). Both duplicate the regex check and SQL keyword validation. If one is updated the other may not be. Fix: Extract the shared validation into a private assertCodeFormat(String code) method and call it from both overloads.
Bug console.log left in production code (api.ts)
api.ts in saveForm() has: console.log("SENDING PAYLOAD TO DB:", ...). This logs the entire form rules JSON to the browser console in production, leaking internal schema. Fix: Remove or gate behind a process.env.NODE_ENV === 'development' check.
Bug FormSchema.id typed as number but backend uses UUID
In schema.ts, FormSchema.id is typed number, but the backend returns a UUID string. Similarly FormSummary.id in page.tsx is number. This works only because JavaScript is loose with string/number coercion in URL building, but TypeScript throws no warning. Fix: Change id: number to id: string everywhere in frontend types.
Bug WorkflowController catches RuntimeException for auth errors instead of using @ResponseStatus
The approve() endpoint catches RuntimeException and manually checks message.contains("Unauthorized") to return 403. This is fragile — a message change silently breaks authorization. Fix: Throw a typed exception (e.g. AccessDeniedException) from WorkflowService and let GlobalExceptionHandler handle it with a 403 response.
Bug hardDeleteForm uses System.err.println instead of proper logging
FormController.hardDeleteForm() catches exceptions with System.err.println() and e.printStackTrace(). These bypass the logging framework and won't appear in structured log output. Fix: Replace with log.error("Permanent delete failed for form {}: {}", id, e.getMessage(), e) using SLF4J.
Bug Timestamp timezone inconsistency across schema
users, audit_logs, workflow_instances/steps, and level_up_requests use TIMESTAMP WITHOUT TIME ZONE, while forms, form_versions, and form_submission_meta use TIMESTAMP WITH TIME ZONE. This causes silent timezone conversion bugs when comparing timestamps across tables. Fix: Migrate all timestamp columns to TIMESTAMP WITH TIME ZONE.

Quality of Life Improvements

Not breaking, but would meaningfully improve developer or user experience

QoL Add SLF4J logging throughout the backend services
The backend currently has zero structured logging in services. All debug output uses System.err.println() (found in 3 places) or nothing. Recommendation: Add @Slf4j (Lombok annotation) to all services and replace println calls. Log at DEBUG for happy path, WARN for business rule violations, ERROR for unexpected exceptions. This is essential for production debugging.
QoL Extract API_SERVER to environment variable in frontend
apiConstants.ts hardcodes const API_SERVER = 'http://localhost:8080'. Deploying to any environment requires code changes. Fix: Change to const API_SERVER = process.env.NEXT_PUBLIC_API_URL ?? 'http://localhost:8080' and add NEXT_PUBLIC_API_URL to .env.local and deployment configs.
QoL Add database indexes for frequently-queried columns
The schema has no explicit indexes beyond PKs and unique constraints. High-frequency queries that would benefit from indexes: form_submission_meta(form_id), workflow_instances(creator_id), workflow_steps(instance_id, status), audit_logs(actor, created_at), user_form_roles(user_id). Add these to schema.sql.
QoL usePermissions cache should survive page navigation
The permissionsCache in usePermissions.ts is a module-level variable — it persists across component mounts but is reset on full page reload. Every page currently calls usePermissions() independently, causing a waterfall of permission checks on initial load. Recommendation: Lift the permission state into a React Context provider at the app root so it is fetched once and shared across all pages without re-fetching.
QoL FormRenderer is a 65KB monolith — split by field type
FormRenderer.tsx at ~65KB is the largest file in the project and handles rendering for all 21 field types, rule evaluation, pagination, draft saving, and file uploads in one component. This makes it hard to test and maintain. Recommendation: Extract each field type into a fields/ subfolder (e.g. fields/TextField.tsx, fields/GridField.tsx) and keep FormRenderer.tsx as an orchestrator. Similarly PropertiesPanel (40KB) should be split by field type sections.
QoL Add migration scripts folder instead of drop-and-recreate
Currently the only way to update the schema is to run unseed.sql then schema.sql again — destroying all data. Recommendation: Adopt Flyway or Liquibase. Add a sql/migrations/ folder with numbered scripts (e.g. V1__initial_schema.sql, V2__add_indexes.sql). This is critical before going to production with real data.
QoL seeder.sql and README reference different role/user names
The README documents roles as ADMIN, MENTOR, INTERN and users as mentor1/intern1. The actual seeder.sql seeds ADMIN, ROLE_ADMIN, USER, BUILDER with only one user (admin). The README is stale. Fix: Update README to reflect actual seeder contents. Optionally add a proper mock_users_seeder with diverse test accounts per role.

Security Issues

Items requiring attention before production deployment

Security CRITICAL: SMTP password hardcoded in application.properties
application.properties contains a real Gmail app password: spring.mail.password=hoid kail kuoy hdod. This is committed to source control and visible to anyone with repo access. Fix immediately: Move to SPRING_MAIL_PASSWORD environment variable. Use ${SPRING_MAIL_PASSWORD} in properties. Add application.properties to .gitignore and use application.properties.example instead.
Security DB password hardcoded: spring.datasource.password=root
Same issue — the database password root is hardcoded. Move to SPRING_DATASOURCE_PASSWORD env var. For production, use a strong random password and restrict the PostgreSQL user to only the formbuilder2 database.
Security File upload endpoint is fully public with no auth
/api/v1/upload and /api/v1/files/** are in the permitAll() list. Anyone on the internet can upload files or enumerate/download uploaded files without any authentication. Fix: Require authentication for upload. For file access, validate that the requesting user has access to the form the file belongs to, or use time-limited signed URLs.
Security CORS allows only localhost:3000 — needs environment config
SecurityConfig hardcodes allowedOrigins("http://localhost:3000"). In production this needs to be the actual frontend domain. Fix: Move to CORS_ALLOWED_ORIGINS environment variable: configuration.setAllowedOrigins(Arrays.asList(corsAllowedOrigins.split(","))).

Unused / Underused Code

Things found that currently add no value, or could add value if wired up

Unused system_configurations table — exists but nothing reads it
Table #17 system_configurations has a repository (SystemConfigurationRepository) and an entity, but no service or controller uses it. Suggestion to use: This is a perfect place to store configurable settings currently hardcoded in application.properties — max fields per form, session timeout, allowed file types — making them adjustable at runtime via an admin UI without redeploy.
Unused roles.parent_role_id — hierarchy defined but never traversed
The roles table has a parent_role_id self-referencing FK enabling hierarchical roles. No service currently traverses this hierarchy — permissions are checked only on the direct role. Suggestion: If you want BUILDER to inherit USER permissions automatically, implement permission resolution that walks the parent chain. Otherwise, remove the column to keep the schema clean.
Unused permissions.feature_id — all rows have 'ALL', never scoped
Every seeded permission has feature_id = 'ALL'. The column was presumably intended to scope permissions to specific features (e.g. only the forms module). Since it's never used for filtering, it adds confusion. Suggestion: Either implement feature-scoped permission checks or remove the column. If keeping it, document its intended use.
Unused DraftService — separate service but minimal use
DraftService.java exists as its own service but has minimal functionality. Draft saving is largely handled inside SubmissionService via the is_draft flag and status='RESPONSE_DRAFT'. Suggestion: Consolidate draft logic fully into SubmissionService and remove the thin DraftService to reduce confusion about which service handles drafts.
Unused forms.issued_by_username — column with no clear owner
The forms table has both creator_id (FK to users), created_by (VARCHAR username), and issued_by_username (VARCHAR). Three fields capturing similar provenance information creates confusion about which one is authoritative. Suggestion: Standardize on creator_id (the FK) and remove the redundant VARCHAR columns, or at least document each field's distinct meaning.
Unused useUIStore — only one boolean, consider removing
useUIStore.ts contains a single isSidebarOpen boolean. This is so minimal it could just be local useState in the layout component, or merged into the main store. The separate store adds indirection for little benefit. Suggestion: Move to useState in the sidebar component or add more UI state to justify keeping it as a dedicated store.

Performance Improvements

Opportunities to improve speed and efficiency

Perf spring.jpa.show-sql=true should be disabled in production
application.properties has spring.jpa.show-sql=true and hibernate.format_sql=true. This logs every SQL query to the console, causing significant I/O overhead in production. Fix: Set spring.jpa.show-sql=false and only enable it in a dev profile (application-dev.properties).
Perf No N+1 guard on WorkflowController.mapInstanceToDetailedMap()
mapInstanceToDetailedMap() accesses instance.getSteps(), and inside each step accesses s.getApprover().getUserFormRoles(). If the LAZY-loaded collections aren't eagerly fetched in the query, this triggers N+1 queries per instance. With 50 workflow instances this could fire hundreds of queries. Fix: Add a JPQL query in WorkflowInstanceRepository with JOIN FETCH for steps and approver roles.
Perf Dashboard fetches forms and stats in two separate sequential requests
The dashboard page fires GET /forms and GET /forms/stats sequentially inside separate useEffect calls. Fix: Fire them in parallel with Promise.all([fetchForms(), fetchStats()]). This halves the perceived load time on the dashboard page.
Perf CalculationService re-parses expressions on every evaluation
CalculationService parses and evaluates calculation formulas on every form submission. If a form has many CALCULATED fields and receives high submission volume, this is wasteful. Suggestion: Cache parsed expression trees per formVersionId + fieldKey using a ConcurrentHashMap to avoid re-parsing identical formulas.
